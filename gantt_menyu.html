Dependencias principales identificadas

INFRAESTRUCTURA
Monorepo (S1) bloquea todos los demas grupos. Sin monorepo no hay backend, frontend ni CI/CD. El schema Prisma (S1-S2) bloquea a todos los modulos de backend.
AUTH → TODO
Auth con JWT y roles (S2-S3) bloquea a SesionMesa, Menu, Pedido, Socket y Pagos. Ningún endpoint de negocio puede construirse sin guards de rol activos.
MENÚ → CARRITO → PEDIDO
El modulo de Menu (S3-S4) bloquea al Carrito (S4-S5). El Carrito bloquea al modulo de Pedido. Sin items disponibles no hay nada que agregar al carrito ni confirmar.
PEDIDO → SOCKET → PAGO
El modulo de Pedido (S4-S5) bloquea a Socket.io (S5-S6), porque los eventos en tiempo real se generan al confirmar un pedido. Socket bloquea a Pagos, ya que el webhook MP emite eventos al cerrar mesa.
PAGO → DIVISION DE PAGOS
Division de pagos (S9-S10) extiende el modulo de Pago MVP. Requiere que el webhook y el cierre de mesa del MVP esten estabilizados para agregar la logica de porciones.
PAGO → FIDELIZACION
El sistema de puntos (S11-S12) se activa al confirmar un pago. Depende del modulo de Pago MVP. El canje de promos depende a su vez de que los puntos esten acumulados correctamente.
FASTAPI → CHATBOT + MOTOR
El microservicio FastAPI (S9-S10) bloquea al Chatbot (S11-S12) y al motor de recomendaciones (S13-S15). Ambos se despliegan sobre el mismo servicio Python.
REDIS → MOTOR RECOMEND.
Redis Pub/Sub (S13-S14) bloquea al motor de recomendaciones (S14-S15). El consumer Python que recalcula los scores escucha eventos de pedido via Redis.
TESTING → BUG FIXING
Cada ciclo de testing genera su propio ciclo de bug fixing inmediatamente posterior. El testing no se agrupa al final: cada modulo se testea y corrige en la misma ventana en que se construye.
QA CARGA → DEPLOY PROD.
Las pruebas de carga y estres (S16-S17) bloquean el deploy de produccion v2.0 (S20-S21). No se lanza a produccion antes de validar rendimiento bajo carga real.
DEPLOY MVP → V2.0
El deploy del MVP en produccion (S8) es prerequisito para los modulos v2.0. Las nuevas funcionalidades se construyen sobre la base productiva del MVP, no sobre staging.
IMPLEMENTACION → DOCUMENTACION
Cada modulo se documenta en la semana siguiente a su implementacion, no al final. La documentacion de schema (S1-S2) y Swagger por endpoint siguen inmediatamente a su construccion.

Descripcion de sprints

S1Fundaciones

Monorepo operativo. Schema Prisma migrado a Supabase. Proyectos Expo y React iniciados. CI/CD a Railway. Documentación del schema. Inicio de la tesis.

S2Auth + Restaurante

JWT, roles y guards. CRUD restaurante y mesas con QR. Panel admin base. App cliente con login e invitado. Tests de auth. Documentación de tipos.

S3QR + Menú

Escaneo QR abre sesión. Menú con categorías, fotos e ingredientes. Admin gestiona catálogo. Tests de sesión y menú. Swagger de auth y menú.

S4Carrito

Carrito con modificación de ingredientes. Precio en tiempo real. Tests de menú/carrito. Corrección bugs de auth y sesión. Swagger de pedidos.

S5Pedido + Socket

Pedido confirmado con snapshot de precio. Socket.io activo: pedido → cocina al instante. Panel Kanban cocina. App mozo. Tests de pedido y socket.

S6Pagos + Dashboard

Pago con MP, cierre de mesa. Dashboard admin con mapa y ventas. Inicio WCAG. Validación webhook MP en staging. Corrección bugs pedido/socket.

S7Estabilización MVP

E2E en dispositivos físicos. Bug fixing crítico. README del monorepo. Build EAS para TestFlight/Firebase.

S8 ★MVP en producción

Deploy completo en Railway. Apps en stores. MVP v1.0 entregado y funcional. Inicio de v2.0.

S9División + Python

División de pagos en backend. Setup FastAPI + SQLAlchemy en Railway. Tablas v2 migradas. Tests de división de pagos. Doc técnica Python.

S10División UI + Python II

UI de división de pagos en la app. FastAPI estable y documentada. Índices PostgreSQL.

S11Fidelización

Puntos con patrón ledger. Chatbot OpenAI. UI puntos/promos. Tests de fidelización. Corrección bugs pagos.

S12Promos + Chatbot UI

Promos de canje con reversión. UI chatbot en app. Panel admin v2. Tests chatbot. Swagger endpoints v2 parciales.

S13Redis + Motor I

Redis Pub/Sub activo. Consumer Python. APScheduler. Tests de chatbot. Doc motor recomendaciones.

S14Motor II + Recomend.

Motor frecuencia + co-ocurrencia con pandas. Endpoint recomendaciones con fallback. Sección "Para vos" en la app. Tests de recomendaciones. Corrección bugs chatbot.

S15Índices + Swagger

Optimización de queries. Swagger completo v2. Tests de motor de recomendaciones. WCAG continuo.

S16Carga + Estrés

Pruebas k6 con 200 usuarios. Estrés WebSockets 50 mesas simultáneas. Auditoría WCAG con VoiceOver/TalkBack. Manual admin y mozo.

S17QA + WCAG

E2E flujo completo v2.0. Auditoría WCAG cierre. Corrección bugs QA/carga. Hardening: rate limit + CORS.

S18Hardening + E2E II

E2E flujo completo v2.0 segunda ronda. Corrección bugs estrés. Monitoring Railway activo. Informe técnico inicio.

S19Deploy v2 producción

Deploy completo v2.0 en Railway. Apps stores v2. Prueba piloto con restaurante real inicio.

S20Piloto + deuda técnica

Prueba piloto segunda semana. Inicio resolución de deuda técnica. Pulido UX. Informe técnico avance.

S21Pulido + informe

Deuda técnica y pulido UX continúan. Inicio del informe final y video demo.

S22Informe + demo

Redacción del informe técnico final. Grabación del video demo. Tesis continúa.

S23Cierre documental

Informe final entregado. Video demo listo. Repositorio ordenado. Preparación presentación.

S24Preparación tribunal

Ensayo de presentación. Ajustes finales del repositorio. Tesis en fase final.

S25 ✓Entrega final

Presentación ante el tribunal. v2.0 completa entregada. Tesis finalizada. Proyecto cerrado.
