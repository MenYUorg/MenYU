Dependencias principales identificadas

INFRAESTRUCTURA
Monorepo (S1) bloquea todos los demás grupos. El schema Prisma (S1-S2) bloquea a todos los módulos de backend.
AUTH → TODO
Auth con JWT y roles (S2-S3) bloquea SesionMesa, Menú, Pedido, Socket y Pagos. Ningún endpoint puede construirse sin guards de rol activos.
MENÚ → CARRITO → PEDIDO
El módulo de Menú (S3-S4) bloquea al Carrito (S4-S5). El Carrito bloquea al módulo de Pedido. Sin ítems no hay carrito ni pedido.
PEDIDO → SOCKET → PAGO
El módulo de Pedido (S4-S5) bloquea Socket.io (S5-S6), que genera eventos al confirmar. Socket bloquea a Pagos, ya que el webhook emite eventos al cerrar mesa.
MENÚ → RECOMENDACIONES
El sistema de recomendaciones (S9) depende del módulo de Menú estabilizado. El dueño solo puede recomendar ítems que ya existen en el catálogo.
PAGO → DIVISIÓN + FIDELIZACIÓN
División de pagos (S10-S11) y el sistema de puntos (S12-S13) extienden el módulo de Pago MVP. Requieren que webhook y cierre de mesa estén estabilizados.
FASTAPI → CHATBOT
El microservicio FastAPI (S10-S11) bloquea al Chatbot (S12-S13). El chatbot se despliega sobre el mismo servicio Python.
QA CARGA → DEPLOY PROD.
Las pruebas de carga y estrés (S15-S16) bloquean el deploy de producción v2.0 (S18-S19). No se lanza a producción antes de validar rendimiento bajo carga real.
DEPLOY MVP → V2.0
El deploy del MVP en producción (S8) es prerequisito para los módulos v2.0. Las nuevas funcionalidades se construyen sobre la base productiva del MVP.
IMPLEMENTACIÓN → DOCUMENTACIÓN
Cada módulo se documenta en la semana siguiente a su implementación. Swagger por endpoint sigue inmediatamente a su construcción.

Descripción de sprints

S1Fundaciones

Monorepo operativo. Schema Prisma migrado a Supabase. Apps React+Vite iniciadas. CI/CD a Railway. Documentación del schema. Inicio de la tesis.

S2Auth + Restaurante

JWT, roles y guards. CRUD restaurante y mesas con QR. Panel admin base. App cliente con login e invitado. Tests de auth.

S3QR + Menú

Escaneo QR con html5-qrcode abre sesión. Menú con categorías, fotos e ingredientes. Admin gestiona catálogo. Swagger de auth y menú.

S4Carrito

Carrito con modificación de ingredientes. Precio en tiempo real. Tests de menú y carrito. Corrección bugs auth y sesión.

S5Pedido + Socket

Pedido confirmado con snapshot de precio. Socket.io activo: pedido → cocina al instante. Panel Kanban cocina. App mozo.

S6Pagos + Dashboard

Pago con MP, cierre de mesa. Dashboard admin con mapa y ventas. Inicio WCAG. Validación webhook en staging.

S7Estabilización MVP

Testing E2E en browsers (Chrome, Safari, Firefox). Bug fixing crítico. README del monorepo. Revisión seguridad y CORS.

S8 ★MVP en producción

Deploy completo en Railway. MVP v1.0 funcional y accesible en producción. Inicio de v2.0.

S9Método de recomendaciones

Sistema simple controlado por el dueño. Tabla Recomendacion con vigencia temporal. CRUD admin con filtros por ingrediente/categoría. Sección Recomendaciones al inicio del menú. Tests.

S10División + Python

División de pagos en backend. Setup FastAPI + SQLAlchemy en Railway. Tablas v2 migradas. Tests de división de pagos.

S11División UI + FastAPI

UI de división de pagos en la app. FastAPI estable y documentada. Doc técnica del microservicio Python.

S12Fidelización + Chatbot

Puntos con patrón ledger. Chatbot OpenAI con context injection. UI puntos/promos. Tests fidelización.

S13Promos + Chatbot UI

Promos de canje con reversión. UI chatbot en app cliente. Panel admin v2. Tests chatbot. Swagger endpoints v2 parciales.

S14Índices + Swagger

Optimización de queries con EXPLAIN ANALYZE. Índices PostgreSQL. Swagger completo v2. WCAG continuo.

S15Carga + Estrés + WCAG

Pruebas k6 con 200 usuarios. Estrés WebSockets 50 mesas simultáneas. Auditoría WCAG con VoiceOver/TalkBack. Manual admin y mozo.

S16QA + Hardening

E2E flujo completo v2.0. Hardening: rate limit + CORS. Bug fixing QA y carga. Manual de usuario admin y mozo.

S17Hardening + E2E II

E2E segunda ronda: escenarios edge case. Monitoring Railway activo. Revisión de logs estructurados. Inicio informe técnico.

S18Deploy v2.0 producción

Deploy completo v2.0 en Railway. Smoke test en producción. Inicio prueba piloto con restaurante real.

S19Piloto + Deuda técnica

Prueba piloto segunda semana. Resolución de deuda técnica crítica. Pulido UX. Avance informe técnico.

S20Pulido + Informe

Deuda técnica y pulido UX continúan. Revisión de animaciones. Avance significativo del informe técnico.

S21Informe + Demo

Completar informe técnico final. Grabación del video demo. Edición. Tesis en fase avanzada.

S22Cierre documental

Entrega informe final. Video demo publicado. Repositorio ordenado. /docs completo. Preparación presentación tribunal.

S23Preparación tribunal

Slides de presentación. Ensayo mínimo 2 rondas. Ajustes finales del repositorio. Tesis en revisión final.

S24 ✓Entrega final

Presentación ante el tribunal. v2.0 completa en producción. Tesis finalizada y entregada. Proyecto cerrado. 🎉
